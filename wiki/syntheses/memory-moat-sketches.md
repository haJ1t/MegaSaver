# Memory Moat Sketches

## i7 [30.3] Mistake Firewall (internal: guard) — PreToolUse failed-attempt interception with priced replay prevention
ARCH: NAMING: market as "Mistake Firewall", internal module name `guard` — "firewall" already means secret-redaction in this repo (packages/pro-analytics/src/firewall-report.ts, redactWithFindings in brain-export).

DATA MODEL (packages/core):
1. NEW `guard-match.ts` — pure, deterministic matcher (no I/O). Input: {toolName, toolInput (command string or file paths), projectId, failures: FailedAttempt[], sessionFailures: SessionFailure[], asOf}. Three tiers, first hit wins:
   - T1 EXACT: normalized-command hash match (whitespace/flag-order/env-prefix normalization) against recorded failing commands, unresolved, age < 30d → deny-capable.
   - T2 PATH: Edit/Write/MultiEdit tool → toolInput.file_path intersects failure.relatedFiles AND BM25 score > 0 over edit context → warn only. Requires BOTH signals (path + text) to fire.
   - T3 BM25: rankBm25 (reuse existing searchFailedAttempts scoring surface: task+failedStep+errorOutput+suspectedCause) over Bash command text, conservative threshold + top-1 margin gate → warn only.
   Resolved failures (resolution set) never deny — they emit positive recall: "solved before: <resolution>".
2. NEW `guard-event.ts` — Zod schema {id, projectId, sessionId, matchedFailureId, tier, action (warn|deny|recall), avoidedTokens, outcome? (heeded|overridden-ok|overridden-failed), createdAt}. New store dir `guard-events/` in json-directory-store (same per-project JSON + atomic tmp+rename pattern).
3. EXTEND SessionFailure capture: the existing PostToolUse saver hook (`mega hooks saver`) already sees every Bash result — on nonzero exit, also record normalizedCommand, exitCode, errorTail, and measured outputTokens (wastedTokens). Consecutive same-command failures in a session sum into one wastedTokens figure. This is where the $ amount comes from — no estimation hand-waving, it's the measured cost of the original failure.

HOOK PATH (apps/cli + connectors/claude-code):
4. NEW CLI subcommand `mega hooks guard` — reads Claude Code PreToolUse JSON from stdin, resolves project by cwd, loads failures (small per-project JSON, target <150ms incl. Node cold start; matcher scoped to Bash/Edit/Write/MultiEdit only — never Read/Grep), runs guard-match, emits hookSpecificOutput: additionalContext ("You tried this 2026-07-11, failed: <errorTail>. Root cause: <suspectedCause>. Original failure cost ~18,200 tokens (~$0.27).") in warn mode, permissionDecision:"deny" + reason in strict mode (T1 only). Appends guard-event. ponytail: no daemon dependency v1; daemon fast-path only if latency measured bad.
5. EXTEND hook-settings.ts — add "guard" to the buildHookCommand subcommand union, new PreToolUse entry with matcher `^(?:Bash|Edit|Write|MultiEdit|NotebookEdit)$`, reuse existing install/uninstall/status/atomic-write plumbing verbatim. Installed automatically by the existing `mega init` connector flow — zero new setup.

FALSE-POSITIVE DEFENSE (the make-or-break):
- Warn-only default; deny (strict mode) restricted to T1 exact-match.
- Per-session cooldown: each failureId fires once per session.
- OUTCOME FEEDBACK LOOP: PostToolUse saver runs on the same tools — after a warn, if the agent ran the command anyway and it succeeded, label the guard-event `overridden-ok` and auto-decay that failure's guard weight (3 strikes → auto-mute + prompt to set resolution). Measured false-positive rate shown in `mega guard status`. This closed loop is the moat detail claude-mem can't copy trivially.
- `mega guard mute <failureId>` manual escape hatch.

CROSS-AGENT (mcp-bridge + connectors):
6. NEW MCP tool `check_approach({description, files?})` — thin wrapper over existing find-similar-failures + memory entries type=failed_attempt via isRecallable; returns matched failures with root cause/resolution, capped output. Connector-managed blocks (Codex/Cursor/Aider via conventions sync) gain one instruction line: "Before non-trivial implementation choices, call check_approach." That is the v1 agent-agnostic story despite PreToolUse being Claude-only.
7. DEFERRED (flag in spec, don't build): llm-proxy passive plan scanner matching streamed plans against the store — v2.

ROI INTEGRATION (packages/pro-analytics):
8. roi.ts/history.ts/insights.ts gain a "retry cost avoided" line summed from guard-events where outcome=heeded, priced from avoidedTokens (= original failure's wastedTokens). forecast.ts includes it; export includes guard-events.

RISK LEVEL: HIGH per §12 (connector core path, public CLI flags) → architect + critic chain, worktree, no main edits.
CLI: Zero-setup core flow: hook auto-installed by existing `mega init` / connector sync.

- `mega hooks guard` — internal PreToolUse entrypoint (stdin JSON → hook JSON). Not user-facing.
- `mega guard status` — enabled/installed, mode, intercepts this month, measured false-positive rate.
- `mega guard mode warn|strict` — strict enables deny on T1 exact-match only. Default warn.
- `mega guard events [--limit N] [--json]` — intercept ledger with matched failure, tier, outcome, $ avoided.
- `mega guard mute <failureId>` / `mega guard unmute <failureId>`.
- `mega guard check "<command or description>"` — dry-run the matcher (demo + debugging tool).
- MCP: `check_approach({description, files?})` — cross-agent, instruction-injected via managed connector blocks.
- Existing surfaces gain lines: `mega roi` ("retry cost avoided: $4.12"), `mega savings history/insights`, `mega savings fix` (suggests enabling guard if disabled).
PRO: Interception itself is FREE — it is the new proof surface and a self-marketing upsell impression: every intercept prints the one-shot dollar line "prevented replay of a failure that cost ~$0.27 — cumulative retry-cost avoided: mega roi (Pro)". Free tier: warn-mode interception with full reason + resolution recall, `guard status` counters, `guard check` dry-run, session-failure auto-capture.

Pro (existing `savings-analytics` key — NO new entitlement key, reuses live gating): cumulative "retry cost avoided" in mega roi / savings history / insights / forecast, `guard events` ledger + export, false-positive-rate analytics, strict (deny) mode, and cross-agent `check_approach` unlimited (free tier caps it to failures from the last 7 days).

Why this split sells: the visceral moment (agent stopped mid-mistake) is free so every demo and every teammate screen-share converts; the accumulating dollar total — the thing that justifies $7.99/mo month after month — is Pro. Matches the proven pattern already in production: savings headline free, analytics paid. Every intercept is a priced ad for the subscription.
EFFORT: 3–4 weeks. Week 1: core matcher + guard-event schema/store + `mega hooks guard` + connector hook install (demoable). Week 2: saver-hook SessionFailure enrichment (wastedTokens capture) + outcome feedback loop + mute/mode/status/events CLI. Week 3: check_approach MCP tool + connector instruction line + roi/history/insights integration. Week 4 buffer: HIGH-risk review chain (architect + critic + code-reviewer), latency measurement, docs, gauntlet. Feasibility discount: BM25 search, hook install plumbing, find-similar-failures, and roi math all already exist — this is mostly wiring, not invention.
RISKS: 1. FALSE POSITIVES (existential): a wrong intercept trains the user to ignore the tool in ~3 occurrences. Mitigated by warn-only default, deny restricted to exact-match tier, two-signal requirement for edits, per-session cooldown, and the PostToolUse outcome-labeling loop that auto-decays stale failures and surfaces a measured FP rate. Ship threshold conservative; loosen with data.
2. HOOK LATENCY: Node cold start ~100ms on every Bash/Edit call. Matcher scoped to mutating tools only; measure in week 1; daemon fast-path is the named upgrade if it hurts.
3. CLAUDE-ONLY v1: PreToolUse exists only in Claude Code — contradicts agent-agnostic thesis. check_approach + injected instruction is the cross-agent bridge; llm-proxy scanner deferred to v2. Be honest in marketing ("deepest on Claude Code, available everywhere via MCP").
4. COPYCAT (claude-mem is literally a hooks plugin): the copyable part is the Claude hook; the moat is measured wastedTokens pricing (needs the saver ledger), the outcome feedback loop, cross-agent check_approach, and roi integration — all of which sit on infrastructure competitors don't have.
5. SPARSE STORE COLD START: guard is silent until failures exist. SessionFailure auto-capture (already passive via saver hook) seeds it without any agent discipline; first intercepts appear within days of normal use.
6. NAME COLLISION: "firewall" already means secret redaction internally — internal name `guard` avoids code confusion; marketing keeps "Mistake Firewall".
FIRST: Spec + build the pure core matcher: `packages/core/src/guard-match.ts` (T1 normalized-command exact match + T3 BM25 with conservative threshold, reusing rankBm25 and the searchFailedAttempts text surface) with a table-driven test suite of real failure/command pairs including adversarial near-misses (same command different repo state, resolved failures, stale failures) — precision is the product, so the matcher tests ARE the spec. Then wire `mega hooks guard` warn-only for Bash and the one-line hook-settings.ts extension. End of slice: record a fake failed attempt, rerun the command in a fresh Claude Code session, watch the warning appear. That is the demo, one week in.
WOW: Split-screen demo, two sessions. Session 1: agent runs `pnpm vitest --shard 2`, fails, burns ~18k tokens flailing, failure auto-captured (user did nothing). Session 2, fresh context, agent about to run the same command — terminal prints: "⛨ Mistake Firewall: you tried this on 2026-07-11 — failed: unknown option '--shard'. Root cause: vitest wants --shard=1/2. That failure cost ~18,200 tokens (~$0.27)." The agent corrects on its very first attempt. No command was typed, no memory was queried — the product reached into the agent's hand mid-mistake and stopped it, with a dollar amount attached. Then one line: "Retry cost avoided this month: $4.12 — mega roi". Every competitor's demo is "ask memory and it answers"; this demo is the agent being saved from itself, unprompted. That's the clip that converts skeptics into $7.99/mo subscribers.

## i8 [29.3] Warm Start — budgeted session boot brief for every agent
ARCH: NO new package. One pure assembler module + thin delivery adapters over existing plumbing.

**1. Core assembler — `packages/core/src/warm-start.ts`** (pure, deterministic, no I/O):
`assembleWarmStartBrief(input: WarmStartInput): WarmStartBrief` where WarmStartInput = { projectId, branch, now, budgetTokens (default 2000), lastSessionEndedAt: string|null (from sessions.json `endedAt`, packages/core/src/session.ts:23), memories: MemoryEntry[], rules: ProjectRule[], failedAttempts: FailedAttempt[], graph: MemoryGraph|null, gitDelta: { commits: {sha,subject,date}[], changedFiles: {path,churn}[] } | null }. The CLI layer gathers inputs (registry reads + two git commands: `git log --since=<lastSessionEndedAt> --format=...` and `git diff --stat <merge-base>`); the assembler stays testable with fixtures.

Filtering reuses the ONE centralized predicate: only `isRecallable(m, now)` memories (approved + current + non-archival — fixes the "inject stale garbage" failure mode for free). Ranking reuses `effectiveConfidence(now)` from memory-entry.ts; rules via `project-rule-ranking.ts`; failed attempts via `searchFailedAttempts` (excludes convertedToRule) then filtered to `relatedFiles ∩ gitDelta.changedFiles` — the branch-aware section competitors don't have. Entity digest = top-5 nodes by mention count from `@megasaver/memory-graph` buildGraph.

**Budget algorithm**: fixed section priority — (1) header: project, branch, last-visit age; (2) current project rules; (3) standing decisions (type=decision); (4) open todos (type=todo, !stale); (5) failed attempts touching branch-changed files; (6) git delta digest; (7) entity digest. Greedy fill: render item → `estimateTokens` (reuse packages/output-filter/src/tokens.ts, ~4 bytes/token) → stop section at cap, stop brief at budget. Every item is one line: `[type] title — first-sentence clamp (confidence, age)`. Output is markdown + a `WarmStartBrief` struct { text, tokenEstimate, mode, sectionCounts } for analytics.

**Modes** (auto-selected from `now − lastSessionEndedAt`): `micro` (<4h: header + rules + open todos only, ~300 tokens — no daily spam), `standard` (4h–14d: full brief), `reonboard` (>14d: budget reallocated to "project state as of your last visit" — expanded git delta grouped by scope, memories/rules that EXPIRED or were superseded/archived in the gap (validTo/expiresAt fell inside the window — first real consumer of the bi-temporal fields, weakness #1), rules added since). Thresholds hardcoded with ponytail comments; only budget is a flag.

**2. Claude Code delivery — SessionStart hook**: extend `packages/connectors/claude-code/src/hook-settings.ts` with a `SessionStart` entry (identical CommandHook shape as existing PreToolUse/PostToolUse/UserPromptSubmit, `buildHookCommand("warmup")`, timeout 10s); `mega hooks install` writes it. New CLI entrypoint `apps/cli/src/commands/hooks/warmup.ts`: reads Claude Code's SessionStart JSON on stdin ({session_id, cwd, source}), resolves project by cwd, assembles, prints brief to stdout (Claude Code injects stdout into context). HARD RULE: fail open — any error ⇒ empty stdout, exit 0. A crashing SessionStart hook would block every session; this is the one place error handling is not optional.

**3. Codex/Cursor/Aider delivery — sentinel block**: new `MEGA_SAVER_WARM_START_BLOCK` rendered by the exact `upsert.ts` mechanism that maintains the context-gate block (packages/connectors/shared/src/context-gate-block.ts is the template to copy). Written by `mega warmup --write` and refreshed by `mega connector sync`. Because block content is written at sync time (not session time), the block carries only the timeless sections (rules, decisions, todos, failed attempts) plus an `As of: <timestamp> — run \"mega warmup --write\" to refresh` line; live git delta stays hook-only. Optional: daemon refreshes the block on HEAD change (defer to v2).

**4. Savings measurement (the thesis)**: new event kind `warm_start` in `packages/stats/src/event.ts` schema, appended via existing `appendEvent`. HONEST method, not asserted 50k: PreToolUse telemetry (`mega hooks log`) already records early-session tool calls. Define cold-start exploration = estimated tokens of Read/Grep/Glob/WebFetch results in the first 20 tool calls of a session. Baseline = median over the project's pre-warm-start sessions already in the stats store. Saved = max(0, baseline − actualEarlyExploration − briefTokens). Fewer than 3 cold sessions on record ⇒ record briefTokens as cost, savings "unmeasured" (honest-metrics.ts precedent). Warm-start line items then flow into the existing savings headline / history / insights / roi with zero extra work.

**MCP**: one optional `get_warm_start_brief` tool in mcp-bridge for agents that poll instead of hook. Thin wrapper over the same assembler.
CLI: `mega warmup` — print the brief to stdout (markdown). Flags: `--budget <tokens>` (default 2000, min 300, max 8000), `--mode auto|micro|standard|reonboard` (default auto), `--json` (WarmStartBrief struct), `--write [--target claude-code|codex|cursor|aider|all]` (Pro: upsert sentinel block into the connector-managed agent files).

`mega hooks install` — now also writes the SessionStart entry (opt out: `--no-warmup`). `mega hooks uninstall` removes it. `mega hooks status` shows it.

`mega hooks warmup` — hook entrypoint only (stdin JSON → brief on stdout, fail-open). Not for humans.

`mega savings` / `mega savings history|insights|roi` — warm-start rows appear automatically as event kind `warm_start` (no new flags).
PRO: Free: the Claude Code SessionStart brief itself, default 2k budget, `mega warmup` stdout, and the savings-headline line "Warm start saved ~Xk tokens this week". Rationale: judges are right that claude-mem primes sessions for free — the brief must be table stakes and a proof surface, not a paywall. It is also the best top-of-funnel demo the product has.

Pro ($7.99, existing keys — no new entitlement): (a) cross-agent injection `mega warmup --write` for codex/cursor/aider under the existing `brain-portability` key (it IS brain-to-every-agent; the only users who need it run multiple agents, which is exactly the segment for whom "better than free claude-mem" converts); (b) reonboard mode's expired/superseded diff + per-session warm-start attribution in history/insights/forecast/roi under the existing `savings-analytics` key. Free users see the reonboard header with an upsell line instead of the expanded body. This split sells because the free tier proves the value daily on Claude Code, and the moment a user opens Codex or Cursor the one command that carries the same brief over is the purchase trigger.
EFFORT: 2 weeks. Week 1: spec+plan (process discipline), TDD the pure assembler + modes + budget enforcement, `mega warmup`, SessionStart hook install/entrypoint, smoke on real Claude Code session. Week 2: sentinel block for other agents, `warm_start` stats event + baseline math, reonboard mode, Pro gating, dual review (connector core path + hook = HIGH risk per §12: code-reviewer AND critic), gauntlet. S-effort core (~500 LOC product code) but HIGH-risk process overhead is real.
RISKS: (1) A SessionStart hook that crashes or hangs blocks every Claude Code session — worst possible product experience; mitigated by fail-open (empty stdout, exit 0), 10s timeout, `--no-warmup`. (2) Context pollution: a stale or wrong brief injected daily actively harms agent performance; mitigated by isRecallable-only content, hard budget, micro mode for <4h gaps — but approval backlog rot (weakness #3) means agent-captured knowledge never reaches the brief, muting its value until that's fixed. (3) Savings honesty: median-baseline methodology is contestable and can read negative; never claim <0, label "estimated", require ≥3 cold sessions — overclaiming here poisons the product's entire proof thesis. (4) Weak moat, acknowledged: claude-mem and Anthropic native memory converge on session priming; the defensible parts are budget discipline, branch-aware failed attempts, cross-agent delivery, and measured savings — position it as table stakes that feeds the analytics moat, not as the differentiation story. (5) Sentinel-block briefs go stale between syncs for non-hook agents; mitigated by timeless-sections-only + as-of stamp, daemon auto-refresh deferred. (6) JSON store full-read per session start is fine at hundreds of memories, degrades at thousands (weakness #5) — acceptable now, noted ceiling.
FIRST: Write the spec (`docs/superpowers/specs/2026-07-12-warm-start-design.md`, risk HIGH), then TDD the pure assembler: failing Vitest suite for `assembleWarmStartBrief` in `packages/core` with fixtures covering — budget never exceeded (estimateTokens ≤ 2000 across adversarial long memories), section priority order, only isRecallable memories included, failed-attempt section empty when no relatedFiles intersect changed files, mode auto-selection at the 4h/14d boundaries, deterministic output for identical input, and reonboard mode surfacing a memory whose validTo fell inside the absence window. No I/O in any test — the assembler is a pure function, so this slice is spec-able and green-able in isolation before any hook or CLI work.
WOW: The demo: open a repo you haven't touched in 6 weeks, type `claude`, and before your first prompt the agent already knows — "Since your last visit (41 days): 23 commits, auth module rewritten; rule 'use repository pattern for stores' still current; your todo 'migrate billing to v2 API' is open; do NOT retry pdfjs for PDF parsing (failed 3x, see failed-attempt fa-19); 2 decisions expired while you were away." Zero re-exploration, zero re-explaining. Then `mega savings` shows "Warm start saved 41k tokens this week (~$1.90)" — measured, not asserted. Close with one command, `mega warmup --write --target all`, and Codex, Cursor, and Aider open with the same brief from the same brain. That last 10 seconds — one brain briefing four agents with a provable dollar figure attached — is the moment a multi-agent dev subscribes.

## i6 [29] Code-Truth Verify — git-anchored memories that stale and heal
ARCH: DATA MODEL (packages/core/src/memory-anchor.ts, new; + 2 optional fields on memoryEntrySchema in memory-entry.ts — additive, legacy rows parse untouched, backfill unchanged):
- `codeAnchorSchema`: { repoHead: sha, capturedAt: ISO, files: [{ path, blobSha }], symbols: [{ path, name, startLine, endLine, contentHash }], deps?: [{ name, claimedVersion }] }. Stored as `anchor?` on MemoryEntry.
- `lastVerified?`: { headSha, at, result: "verified"|"contradicted"|"healed" } for badges. Contradiction reuses EXISTING fields: flip `stale: true`, close `validTo: now`, append machine-written string to `evidence[]` ("code-truth: contradicted by commit 4f2a91c — src/auth/middleware.ts#verifyToken removed"). Healing reverses: `stale: false`, `validTo: null`, healing evidence appended. Never deletes — feeds the write-orphaned bi-temporal fields (weakness #1) with real data for free.

CAPTURE (write path): async helper `captureCodeAnchor(rootPath, relatedFiles, relatedSymbols)` in core, called by CLI `memory create/update`, MCP `save_memory`, daemon — BEFORE the sync `registry.createMemoryEntry` (registry signature stays sync; anchor rides in on the entry). Per file: `git -C rootPath rev-parse HEAD:path` blob sha. Per symbol: run the EXISTING `@megasaver/indexer` extractors (extract-ts/py/go/rs) on the cited file, match block by name, copy its contentHash + span. Best-effort: any failure ⇒ save proceeds unanchored. Project.rootPath already exists, so verify knows where to run git.

VERIFY ENGINE (packages/core/src/code-truth.ts): pure/impure split matching sweepMemoryTiers style. Pure planner `verifyAnchors(entries, repoState) → { contradicted: [{id, reason}], healed: [id], verified: [id] }` — unit-testable with injected repoState, zero git in tests. Impure runner: one `git rev-parse HEAD`, one batched `git cat-file --batch-check` for all file blobs, re-extract only files cited by symbol anchors. Contradiction policy (the false-stale defense): file-blob change ALONE is never a contradiction; contradiction = file deleted, cited symbol missing, or symbol contentHash changed. Missing file first tries rename detection (`git diff --name-status -M <anchorHead>..HEAD -- <path>`) and self-repoints the anchor instead of flagging. Attribution: `git log -n1 --format=%H <anchorHead>..HEAD -- <path>` names the falsifying commit. Dependency checker: memories of type `dependency` with a name@version pattern compared against package.json (~40 lines). Mutations via existing `registry.updateMemoryEntry`.

TRIGGER SURFACES: (1) one-shot `mega memory verify`; (2) git post-commit hook running `verify --changed --quiet` scoped to `git diff-tree --name-only HEAD` (sub-second), installed with the same sentinel-block idempotent-write pattern the connector sync already uses; (3) folded into `mega memory sweep` as a pre-pass so contradicted rows archive in the same run — sweep already archives `entry.stale`, so ZERO new sweep logic; (4) pre-recall spot-check in MCP `get_relevant_memories`: top-5 anchored hits, mtime-vs-capturedAt pre-filter then re-hash, ~50ms budget, contradicted hits down-ranked + tagged `contradicted-by-code` in the payload, flip persisted async.

RECALL/RANKING: add `STALE_WEIGHT = 0.3` multiplier to `effectiveConfidence` for stale rows still in recall tier (down-rank, never drop — matches existing decay philosophy). GUI: badge chips (verified/contradicted/unanchored) on MemoryPanel + header count "3 memories contradicted by current code" — additive fields through the existing GET /api/memory bridge. Savings: each spot-check demotion logs an "avoided stale recall" event (token size of the demoted memory) into the existing savings pipeline as a new insights source line. Brain export/import: anchors are repo-relative shas — they flow through .megabrain automatically (schema-additive) and verify Just Works on the destination machine. No LLM anywhere; CI-safe; risk level HIGH per §12 (memory schema change) ⇒ architect + critic passes required.
CLI: `mega memory verify [--changed] [--quiet] [--json] [--project <id>]` — full or diff-scoped pass; prints "3 contradicted, 1 healed, 41 verified, 12 unanchored" table, per-row reason + falsifying commit.
`mega memory verify --install-hook | --uninstall-hook` — idempotent .git/hooks/post-commit block (Pro).
`mega memory sweep` — gains verify pre-pass; `--no-verify` opts out.
`mega memory create/update` — auto-capture anchor when relatedFiles/relatedSymbols present; `--no-anchor` opts out.
`mega memory show <id>` — prints anchor, verification badge, evidence trail.
MCP: `save_memory` captures anchors; `get_relevant_memories` returns `verification` badge + spot-check demotion; new `mega_memory_verify` tool (thin alias over the engine).
PRO: FREE: anchor capture always-on (the data must accumulate for everyone or later conversion has nothing to verify), one-shot `mega memory verify`, and GUI badges. This is the trust/proof surface — the user SEES "3 memories contradicted by current code" for free, which is the moment every claude-mem/mem0 user realizes their current tool serves them lies.
PRO (new ProFeature `"code-truth"` in the existing entitlement union; checkEntitlement already feature-agnostic, gate-first + upsell-print pattern copied from brain commands): post-commit hook automation, sweep-integrated auto-verify, pre-recall spot-check guard, heal/contradiction audit history, and the "stale recall waste avoided: $X" line in savings insights (which sits under savings-analytics anyway).
Why the split sells: free tier diagnoses the disease, Pro is the immune system — "see the rot free; pay $7.99 so wrong memories never reach your agent again, and watch the avoided waste show up in dollars." The upsell trigger is organic: every free verify run that finds contradictions prints the hook upsell.
EFFORT: 3–4 weeks. W1: anchor schema + captureCodeAnchor + create-path wiring (TDD, temp-git-repo integration test). W2: pure verifyAnchors planner + git runner + `mega memory verify` + sweep fold. W3: post-commit hook, pre-recall spot-check, MCP/GUI badges. W4: savings event, rename-mitigation polish, HIGH-risk review gauntlet (architect + code-reviewer + critic per §12).
RISKS: (1) False-stale noise from refactors/renames — the judge's flagged risk. Mitigations baked in: symbol-hash (not file-hash) is the contradiction unit, file-content change alone never contradicts, `git diff -M` rename detection self-repoints anchors, and healing makes every false flip recoverable and auditable rather than destructive. (2) Worktree-vs-HEAD truth ambiguity: spec must pin one semantics (recommend: verify against worktree content, attribute via last commit touching the path); dirty-tree edge cases otherwise flap. (3) `updatedAt` reset on stale-flip inflates the decay freshness of a wrong memory — STALE_WEIGHT + sweep-to-archival dominate, but spec should note it (interacts with known weakness #4). (4) Perf on large repos/memory sets — batched cat-file, --changed scoping, 50ms spot-check budget with mtime pre-filter; per-project JSON store ceiling (weakness #5) unchanged but not worsened. (5) Cursor could ship git-anchored staleness for Cursor within a quarter — moat is the cross-agent local store + bi-temporal audit trail + $-savings proof, none of which a single-editor clone gets. (6) Non-git projects degrade to unanchored gracefully — must not error.
FIRST: Spec `codeAnchorSchema` + the pure `verifyAnchors(entries, repoState)` planner in packages/core (mirroring the existing sweepMemoryTiers pure-planner pattern), with fixture-injected repoState so unit tests need no git; plus one temp-git-repo integration test for `captureCodeAnchor`. That slice is fully spec-able tomorrow, touches no existing behavior, and every later surface (CLI, hook, sweep, spot-check) is a thin caller of it.
WOW: The live demo loop, 30 seconds, no LLM: save a memory citing `src/auth/middleware.ts#verifyToken`; refactor deletes the function; `git commit` — the post-commit hook instantly prints "1 memory contradicted by commit 4f2a91c: verifyToken removed"; the agent's next recall returns that memory flagged `contradicted-by-code` instead of asserting it as truth; then `git revert` and the hook prints "1 memory healed — hash matches again," with the whole falsify/heal trail sitting in evidence[] and closed/reopened validTo. No other memory tool on the market — claude-mem, mem0, Zep, Letta — can even detect that its memory just became a lie, let alone name the commit that falsified it and heal on revert.
