# II — GUI App Bootstrap — Plan

Bootstrap a `apps/gui` workspace package: Vite + React SPA frontend
+ tiny `node:http` bridge that imports `@megasaver/core`. Two views
(Sessions, Memory), one smoke test, one closed-enum tuple-pin.

## Steps

1. **Worktree confirmed**
   Branch `feat/ii-gui-app` off `origin/main` (`480ec8c`).
   Worktree at `.worktrees/ii-gui-app`.

2. **Scaffold `apps/gui` package**
   Create:
   - `apps/gui/package.json` — name `@megasaver/gui`, private, ESM.
     Dev deps: `react`, `react-dom`, `@types/react`, `@types/react-dom`,
     `vite`, `@vitejs/plugin-react`, `@testing-library/react`,
     `@testing-library/jest-dom`, `jsdom`, `tsx`. Runtime deps:
     `@megasaver/core` (workspace).
   - `apps/gui/tsconfig.json` — extends `../../tsconfig.base.json`,
     adds `"lib": ["ES2023", "DOM", "DOM.Iterable"]`, `"jsx": "react-jsx"`.
   - `apps/gui/tsconfig.test.json` — mirrors CLI's pattern.
   - `apps/gui/tsconfig.test-d.json` — for `.test-d.ts` typecheck.
   - `apps/gui/vite.config.ts` — React plugin, dev server on 5173,
     proxy `/api` → `http://localhost:5174`.
   - `apps/gui/vitest.config.ts` — `environment: "jsdom"`, includes
     `test/**/*.test.{ts,tsx}` + `.test-d.ts`.
   - `apps/gui/index.html` — Vite entry with `<div id="root">` +
     `src/main.tsx` mount.

3. **Closed enum + tuple-ordering pin** (TDD red first)
   - `apps/gui/src/view-id.ts`: `VIEW_IDS = ["memory", "sessions"]
     as const` (alphabetic — matches AA3 convention).
   - `apps/gui/test/view-id.test-d.ts`: tuple-ordering pin using
     `expectTypeOf` and `toEqual` on `[...VIEW_IDS]`.

4. **Bridge implementation** — `apps/gui/bridge/`
   - `store-path.ts`: replicates CLI's store resolution
     (`MEGASAVER_GUI_STORE` env → `XDG_DATA_HOME/megasaver` →
     `$HOME/.local/share/megasaver`). Throws on missing `HOME` —
     matches CLI behavior.
   - `server.ts`: `node:http` server on port 5174 (configurable via
     `MEGASAVER_GUI_BRIDGE_PORT`). Three routes:
     - `GET /api/health` → `{ ok: true, store: <abs path> }`
     - `GET /api/sessions` → flat `Session[]` across all projects
     - `GET /api/memory` → flat `MemoryEntry[]` across all projects
     Uses `JsonDirectoryRegistry` from `@megasaver/core`.
     CORS: not needed (Vite proxy handles same-origin).

5. **React SPA implementation** — `apps/gui/src/`
   - `lib/api-client.ts`: `fetchSessions()`, `fetchMemory()` —
     thin wrappers over `fetch("/api/...")` returning typed arrays
     (`Session[]` / `MemoryEntry[]` from `@megasaver/core` types).
   - `views/sessions-view.tsx`: lists sessions in a `<table>`.
     Columns: id, title, agent, riskLevel, status, createdAt.
   - `views/memory-view.tsx`: lists memory entries in a `<table>`.
     Columns: id, scope, sessionId, content (first 80 chars),
     createdAt.
   - `app.tsx`: view switcher (two buttons), renders current view.
     Local `useState<ViewId>`. Initial view: `"sessions"`.
   - `main.tsx`: React root mount.

6. **Smoke test** — `apps/gui/test/app.test.tsx`
   Mocks `fetch` via `vi.stubGlobal("fetch", ...)` returning `[]`
   for both endpoints. Renders `<App />` with
   `@testing-library/react`. Assertions:
   - both view-switcher buttons render with their labels
     ("Sessions", "Memory entries")
   - default view "Sessions" is active
   - clicking "Memory entries" switches the view (table header
     contains the memory-specific column)
   - empty state message renders ("no sessions" / "no memory
     entries")

## Failure-mode notes

- `@megasaver/core` does not currently expose
  `listAllSessions()` / `listAllMemoryEntries()`. The bridge
  iterates `registry.listProjects()` and flatMaps per-project
  results. This keeps core public surface unchanged.
- Empty store: bridge calls `ensureStoreReady(rootDir)` (already
  exported by core via `initStore`) before listing, so the GUI
  works against an unconfigured machine. Matches `apps/cli`
  initialization pattern.
- If `tsx` is unavailable in CI, the `bridge` script fails. Bridge
  is dev-only; CI never runs it. Document this explicitly in spec
  §Architecture.
