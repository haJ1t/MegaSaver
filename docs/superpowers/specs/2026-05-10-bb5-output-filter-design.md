# BB5 — `@megasaver/output-filter` package

## 1. Scope & authority

The epic spec
`docs/superpowers/specs/2026-05-10-aa1-context-gate-epic.md`
is the authority. This child spec locks the BB5 public surface,
Zod schemas, closed enums, error codes, pipeline order, and the
§3c dependency allow-list. Sections cited: §11 (output filter),
§8a (`resolveSafeReadPath` consumer / hard byte ceiling), §3
(workspace layout, dep graph, cycle guardrails), §9d / §9-BB5
(redaction reconciliation), §17 (tuple pins).

`@megasaver/output-filter` is **HIGH risk** — it is the
secret-leakage choke point. Per epic §12 and §14, a
`security-reviewer` audit is mandatory before merge.

## 2. Mission of the package

One bounded context: **turn a large raw output blob into a small,
high-signal result without losing evidence the agent needs.**

- `filterOutput(input): FilterOutputResult` — **pure, no IO**.
  Runs the locked 9-stage pipeline (§11b).
- `resolveSafeReadPath(input): ResolvedPath` — the **one**
  filesystem-touching helper (calls `fs.realpathSync`). Kept
  deliberately separate from `filterOutput` so callers (BB7a
  `mega output file`, BB8 `mega_read_file`) compose
  `policy.evaluatePathRead` → `resolveSafeReadPath` → `fs.readFile`
  → `filterOutput`.
- Owns two closed enums consumed elsewhere: `RankFeatureName`
  (this package only) and `OutputSourceKind` (consumed by
  content-store §10d and stats §13a; BB5 is the **owner**).

Persistence is the caller's responsibility. `chunkSetId` is
generated by the context-gate orchestrator
(`packages/core/src/context-gate/run.ts`, BB7a) and passed back
into the result when content-store persisted the chunk set; BB5
never calls content-store.

  ownership stays in policy. This is consistent with §3c

  `filterOutput(...)` (not a re-test of policy internals). They

- Redaction-pattern *additions* remain LOW-risk policy PRs with
  the policy corpus; BB5 does not own that corpus.

### 5.1 `filterOutput` signature & I/O schemas (§11a, verbatim)

```ts
// src/types.ts
import { tokenSaverModeSchema, riskLevelSchema } from "@megasaver/shared";

export const filterOutputInputSchema = z.object({
  raw: z.string(),
  intent: z.string().min(1).optional(),
  mode: tokenSaverModeSchema,
  maxReturnedBytes: z.number().int().positive().optional(),
  sessionHints: z.object({
    title: z.string().nullable().optional(),
    recentFiles: z.array(z.string()).readonly().optional(),
    recentMemory: z.array(z.string()).readonly().optional(),
    projectConventions: z.array(z.string()).readonly().optional(),
    risk: riskLevelSchema.optional(),
  }).optional(),
  source: z.discriminatedUnion("kind", [
    z.object({ kind: z.literal("file"),    path: z.string() }),
    z.object({ kind: z.literal("command"), command: z.string(), args: z.array(z.string()).readonly() }),
    z.object({ kind: z.literal("grep"),    query: z.string() }),
    z.object({ kind: z.literal("fetch"),   url: z.string() }),
  ]).optional(),
}).strict();

export type FilterOutputInput = z.infer<typeof filterOutputInputSchema>;

export type OutputExcerpt = {
  text: string;
  startLine: number;
  endLine: number;
  score: number;
  features: RankFeatures;
};

export type FilterOutputResult = {
  summary: string;
  excerpts: readonly OutputExcerpt[];
  rawBytes: number;
  returnedBytes: number;
  bytesSaved: number;
  savingRatio: number;          // 0..1
  chunkSetId?: string;          // set by caller when content-store persisted
  warnings?: readonly string[]; // e.g. "redacted 2 secret(s) before processing"
};

export function filterOutput(input: FilterOutputInput): FilterOutputResult;
```

Boundary validation: `filterOutput` parses `input` with
`filterOutputInputSchema` at entry (§8 — validate at boundaries).
The `.strict()` rejects unknown keys. After parse, internal
stages trust their inputs (no defensive re-checks).

### 5.2 `resolveSafeReadPath` (§11a + §8a, verbatim signature)

```ts
// src/resolve-safe-read-path.ts
export type ResolveSafeReadPathInput = {
  path: string;
  projectRoot: string;          // absolute path, validated by caller
};
export type ResolvedPath = { absolute: string };

export function resolveSafeReadPath(input: ResolveSafeReadPathInput): ResolvedPath;
```

Structural sandbox gate. Throws `OutputFilterError("path_unsafe")`
on any violation:

1. **`..`-traversal** — resolve `path` against `projectRoot`
   (when relative) or use as-is (when absolute); after
   `path.resolve` the result must remain inside one of the
   allowed roots. A `..` segment that climbs above all allowed
   roots is rejected.
2. **out-of-sandbox absolute path** — the resolved absolute path
   must be contained in `projectRoot ∪ cwd ∪ HOME` (prefix match
   on a normalised, separator-terminated boundary so `/home/foo`
   does not match `/home/foobar`). `cwd` is `process.cwd()`;
   `HOME` is `os.homedir()`.
3. **symlink escape** — `fs.realpathSync(resolved)` must still be
   contained in the allowed roots. If the realpath differs from
   the lexically-resolved path AND lands outside the sandbox,
   reject. (A symlink that stays inside the sandbox is allowed.)
   If the target does not exist yet, `realpathSync` on the
   nearest existing ancestor is used for the containment check
   (a read of a non-existent file is the caller's `fs.readFile`
   error, not a sandbox violation).

The allowed-root set is computed once per call from
`{ projectRoot, process.cwd(), os.homedir() }`, de-duplicated and
normalised to absolute. Containment uses `path.relative(root,
target)` not starting with `..` and not absolute.

This is the **only** module importing `node:fs` / `node:path` /
`node:os`. `filterOutput` and all stage modules stay pure.

## 6. Pipeline order (LOCKED — §11b)

`filterOutput` runs exactly these stages in this order. Stage 1
before everything is the **critical security invariant**: secrets
are removed before any value leaves a stage that a caller could
persist.

1. **Redact** — `redact(raw)` from `@megasaver/policy`. If
   `count > 0`, append a warning
   `"redacted <count> secret(s) before processing"`.
2. **Normalize** — strip ANSI escape sequences
   (`\x1b\[[0-9;]*m` and CSI/OSC forms), collapse `\r\n` → `\n`
   and lone `\r` → `\n`, trim trailing whitespace per line.
3. **Collapse repeated lines** — consecutive identical lines
   become one line plus `… [repeated N times]` when `N >= 2`.
4. **Chunk** — default `chunkByLines(40)`. Specialised parsers
   take precedence when their format is detected:
   `test-output`, `ts-diagnostic`, `stacktrace`. Detection is
   first-match by signature; on no match, fall back to line
   chunking.
5. **Rank** — `scoreChunk(intent, chunk, sessionHints)` →
   `RankedChunk` carrying a `RankFeatures` record keyed by
   `RankFeatureName`. The scoring formula subtracts
   `duplicatePenalty` and `noisePenalty` (positive magnitudes;
   sign lives in the formula, not the field name — §11c).
6. **Dedupe** — drop near-duplicate chunks via Hamming distance
   on a 64-bit simhash of normalised chunk text; threshold
   `HAMMING_DEDUPE_THRESHOLD = 3` (pinned in BB5).
7. **Fit byte budget** — greedy pick by descending score until
   adding the next chunk would exceed the effective budget
   (§6.1). UTF-8 byte length via `Buffer.byteLength`.
8. **Summarize** — deterministic templating (NO LLM), mode-keyed:
   `safe` → medium summary, `balanced` → short, `aggressive` →
   tiny. Summary reports counts (chunks kept/dropped, top error
   line if any).
9. **Compose result** — `rawBytes = Buffer.byteLength(raw)`;
   `returnedBytes` = summary + excerpts byte length;
   `bytesSaved = max(0, rawBytes - returnedBytes)`;
   `savingRatio = rawBytes === 0 ? 0 : bytesSaved / rawBytes`.

### 6.1 Effective byte budget (§11d + §8a hard ceiling)

```
effectiveBudget =
  input.maxReturnedBytes ?? modeToBudget(mode)
```

`modeToBudget` is imported from `@megasaver/shared` (single
source of truth: `aggressive→4000`, `balanced→12000`,
`safe→32000`). If `maxReturnedBytes` is present it overrides but
is clamped to the **hard ceiling** `HARD_CEILING_BYTES = 2 *
modeToBudget("safe") = 64_000` (§8a). A `maxReturnedBytes` above
the ceiling is clamped (not rejected) inside `filterOutput`;
callers that want to *reject* an over-ceiling value do so at
their own boundary (BB8 raises `max_bytes_exceeded`).

## 11. Acceptance criteria (from epic BB5)

- Large input → small result (`savingRatio > 0` on a multi-KB
  blob; returned bytes ≤ effective budget).
- Error lines rank to the top (error/diagnostic/stacktrace chunks
  appear in `excerpts` ahead of noise).
- Repeated lines collapsed (`… [repeated N times]`).
- **Property test + corpus test** prove no §9d redaction pattern
  survives `filterOutput` (F-MED-1).
- `resolveSafeReadPath` rejects symlink escapes, `..`-traversal,
  and out-of-sandbox absolute paths; accepts in-sandbox relative
  and absolute paths.
- `dependency-graph.test.ts` passes: deps ⊆ {policy, shared, zod};
  no core.
- All three `*.test-d.ts` tuple pins pass (alphabetic order
  locked).
- `pnpm verify` green across the whole monorepo.

## 12. ADR

**Decision.** Ship `@megasaver/output-filter` as a pure
`filterOutput` pipeline plus a separate IO `resolveSafeReadPath`
gate; import redaction from policy rather than re-owning it.

**Drivers.** (1) Secret-leakage is the dominant failure mode —
redact-first ordering must be unconditional and auditable.
(2) §3c forbids a core dependency to avoid the §2e cycle.
(3) Single source of truth for redaction patterns (already in
policy) and for the mode→budget map (already in shared).

**Alternatives considered.**

- *Re-own redaction patterns in output-filter* (literal older
  epic §9d wording). Rejected: duplicates the policy corpus,
  guarantees drift, contradicts the shipped BB3 surface and the
  §3c "imports policy" graph.
- *Fold `resolveSafeReadPath` into `filterOutput`*. Rejected:
  would make the pure pipeline touch the filesystem and force
  every `filterOutput` caller to supply a real path; the two have
  different callers and different test strategies.
- *LLM-based summarization.* Rejected: non-deterministic, can
  invent or drop evidence, violates the "preserve evidence"
  mission. Deterministic templating only.

**Why chosen.** Minimises surface, respects the dependency graph,
keeps the security choke point pure and independently testable,
and reuses the already-shipped policy redaction with a
pipeline-level invariant test instead of a duplicated pattern
corpus.

**Consequences.** content-store and stats type-import
`OutputSourceKind` from this package (one-directional, no cycle).
The pure/IO split means BB7a/BB8 must compose three calls for a
file read; that composition is specified in §8a and is each
caller's responsibility.

**Follow-ups (out of BB5 scope).**

- When BB4 (`content-store`) lands or rebases onto BB5, delete any
  local `OutputSourceKind` placeholder and type-import from
  `@megasaver/output-filter` (epic §14-bis). Tracked in
  open-questions.
- Redaction-pattern additions ship as LOW-risk policy PRs with the
  policy corpus, not here.
